🚨 Analisis Error

⏱ Semua 36 error terjadi dalam window 10 menit: menit ke-20:50 – 21:00
  • Endpoint: /auction/customer-history-bid (11 error dari 2a, 25 error dari 2j-fresh)
  • HTTP Code: 500 Internal Server Error
  • Elapsed saat error: ~20,000 ms — server menunggu 20 detik sebelum timeout
  • Setelah menit ke-21: 0 error hingga test selesai — server recover sendiri
⚠️ Risiko di Production Threshold error tercapai di menit ke-20 staging. Di production dengan data history_bid yang jauh lebih besar, threshold ini bisa tercapai lebih cepat — estimasi menit ke-5 hingga ke-10. Request yang "menggantung" 20 detik sebelum 500 berpotensi menyebabkan cascading failure.

📌 Temuan & Rekomendasi

🔴 CRITICAL — Wajib Ditangani Sebelum Production Endpoint: GET /auction/customer-history-bid

Dipanggil 2x per bid cycle (step 2a & 2j-fresh). Bottleneck utama sistem.

  • Avg response time 30 menit: ~22,000 ms
  • HTTP 500 muncul di menit ke-20 dengan 100 user concurrent
  • Query makin lambat seiring data history_bid bertumbuh

Rekomendasi:

  1. Tambahkan database index pada kolom auction_id, auction_lane, auction_lot, user_id
  2. Implementasi caching (Redis / in-memory) untuk data yang tidak berubah cepat
  3. Evaluasi apakah query perlu pagination atau limit hasil
🟡 MINOR — Perlu Diperhatikan
  • GET /frontsite/terms-and-conditions — avg 1,707ms. Data statis, kandidat caching.
  • GET /frontsite/list-translation — avg 1,001ms. Data statis, kandidat caching.
