Mensimulasikan lonjakan tiba-tiba (spike) traffic sebesar 100 concurrent users yang masuk dalam waktu 60 detik — mencerminkan skenario realistis saat notifikasi atau broadcast dikirim ke semua user tepat saat auction dimulai.

Mengapa 100u Ramp 60 detik? Skenario 150u Ramp 10 detik (15 user/detik) terlalu ekstrem dan tidak mencerminkan perilaku nyata. 100u dalam 60 detik (~1.7 user/detik) lebih realistis — masih "spike" namun sesuai dengan pola user yang masuk saat mendapat notifikasi auction.

⚠️ Temuan & Rekomendasi

🔴 CRITICAL — Frontsite Endpoints Tidak Tahan Spike

8 endpoint frontsite mengalami HTTP 500 saat 100 user masuk dalam 60 detik:

  • /frontsite/catalog/membership — error 88-89%
  • /frontsite/terms-and-conditions — error 88%
  • /frontsite/option/car-brands — error 67%
  • /auction/auction-list — error 63%
  • /frontsite/catalog/membership/location — error 65%

Root cause:

Semua endpoint ini mengembalikan data yang relatif statis (catalog, T&C, car brands) namun di-query ulang ke database setiap request. Tanpa caching, saat 100 user hit secara bersamaan server langsung kewalahan.

Rekomendasi:

  1. Implementasi HTTP response caching (Redis / CDN / in-memory) untuk data statis
  2. TTL yang disarankan: Car Brands, T&C = 1 jam; Catalog = 5 menit
  3. Audit query database di endpoint-endpoint tersebut — kemungkinan ada N+1 query atau full table scan
✅ Bid Core Terbukti Tahan Spike

Seluruh alur bidding (WebSocket open → join → check latest → bid → ACK) berjalan dengan 0 error meski server sedang dalam kondisi spike. Ini menunjukkan arsitektur bid core sudah solid dan siap menghadapi lonjakan user.

🔴 CRITICAL — History Bid Tetap Menjadi Bottleneck

Sama seperti temuan di Endurance Test, /auction/customer-history-bid memiliki avg 10,133ms saat spike. Meskipun tidak menghasilkan error di test ini (karena hanya 1 loop), di production dengan data yang lebih besar risikonya tetap tinggi.

💡 Catatan untuk PO / Stakeholder

Halaman awal auction (catalog, terms, car brands) adalah halaman yang paling banyak diakses saat user baru masuk — dan justru ini yang paling rentan saat spike. Investasi caching di layer ini akan memberikan dampak terbesar terhadap stabilitas sistem saat auction event berlangsung.
